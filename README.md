# imessage-exporter-viewer

A self-hosted web app for searching and browsing iMessage exports generated by [imessage-exporter](https://github.com/ReagentX/imessage-exporter). Runs entirely in Docker, stores a deduplicated SQLite index, and serves a paginated chat interface with full-text search.

![imessage-exporter-viewer screenshot](https://raw.githubusercontent.com/mbaran5/imessage-exporter-viewer/main/screenshot.png)

## Features

- **Full-text search** across all conversations with porter stemming and prefix matching (`taco` finds `tacos`)
- **Virtual scroll** — conversations with tens of thousands of messages scroll smoothly without pagination UI
- **Inline reply previews** — tapbacks, threaded replies, and edited messages render faithfully from the original exporter HTML
- **Multi-archive deduplication** — index multiple export snapshots without duplicate messages
- **HEIC/HEIF support** — iPhone photos converted to JPEG on the fly for browser display
- **Name resolution** — phone numbers resolved to contact names across group chats
- **Reply navigation** — ⇱ / ⇲ arrows jump between a reply and its parent message
- **Search result highlighting** — clicking a search result jumps directly to the message with a yellow highlight

## Requirements

- [imessage-exporter](https://github.com/ReagentX/imessage-exporter) output directory (one or more export snapshots)
- Docker and Docker Compose

## Quick Start

### Docker Compose (recommended)

1. Clone the repository:
   ```bash
   git clone https://github.com/mbaran5/imessage-exporter-viewer.git
   cd imessage-exporter-viewer
   ```

2. Edit `compose.yaml` to point at your export directory:
   ```yaml
   volumes:
     - /path/to/your/imessage_exports:/archives:ro
     - imessage_db:/data
   ```

3. Start the stack:
   ```bash
   docker compose up -d
   ```

4. Open [http://localhost:6333](http://localhost:6333)

The indexer runs automatically on every container start. On first run it will parse and index all exports — this may take a few minutes depending on archive size. Subsequent starts are fast as messages are deduplicated by GUID.

### Docker Hub

A pre-built image is available:

```bash
docker pull mbaran5/imessage-exporter-viewer:latest
```

```yaml
services:
  imessage-exporter-viewer:
    image: mbaran5/imessage-exporter-viewer:latest
    container_name: imessage-exporter-viewer
    ports:
      - "6333:6333"
    volumes:
      - /path/to/your/imessage_exports:/archives:ro
      - imessage_db:/data
    environment:
      - ARCHIVE_ROOT=/archives
      - DB_PATH=/data/imessage.db
    restart: unless-stopped

volumes:
  imessage_db:
```

### Running locally

```bash
pip install -r requirements.txt

export ARCHIVE_ROOT=/path/to/imessage_exports
export DB_PATH=/path/to/imessage.db

python3 indexer.py
python3 app.py
```

## Exporting your iMessages

Install [imessage-exporter](https://github.com/ReagentX/imessage-exporter) and run:

```bash
imessage-exporter -f html -o ~/imessage_exports
```

Point `ARCHIVE_ROOT` at the output directory. You can export multiple times over the years and point them all at the same `ARCHIVE_ROOT` subdirectory — the indexer will deduplicate across snapshots.

## Archive Layout

The indexer auto-discovers export directories. Supported layouts:

```
/archives/                          # single export at root
/archives/export-2023/              # multiple exports as subdirectories
/archives/exports/2023/             # one level deeper
```

Each discovered directory must contain at least one `.html` file and an `attachments/` subdirectory.

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `ARCHIVE_ROOT` | `/archives` | Root directory containing iMessage HTML exports |
| `DB_PATH` | `/data/imessage.db` | Path to the SQLite database |

## Notes

- Large conversations (40MB+ HTML files) are handled via virtual scrolling — only ~150 messages are in the DOM at a time
- HEIC images are converted to JPEG on the fly; this requires `libheif-dev` (included in the Docker image)
- The FTS index uses the SQLite porter stemmer, so searches are stemmed and prefix-matched automatically
- Re-running the indexer is safe and idempotent — existing messages are not duplicated
